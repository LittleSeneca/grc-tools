# Asset Management Policy — Implementation Procedures

> **Companion to:** Asset Management Policy (AMP-Template.md)
> **Purpose:** These procedures describe how to implement the requirements set forth in the Asset Management Policy. The policy defines WHAT must be done; this document describes HOW to do it.

---

## Procedure 1: System Hardening

### Standard Approach

This procedure establishes the baseline hardening configuration for all organizational systems.

#### 1.1 Hardening Baseline Selection

1. Identify the appropriate hardening benchmark for the system type:
   - **Windows Server / Workstation:** CIS Microsoft Windows Benchmark, Level 1 (or Level 2 for high-security).
   - **Linux Server:** CIS Distribution-Specific Benchmark (RHEL, Ubuntu, Debian, etc.), Level 1.
   - **macOS:** CIS macOS Benchmark.
   - **Network Devices:** CIS Network Device Benchmark or vendor-specific STIG.
   - **Cloud Services:** CIS Benchmarks for AWS/Azure/GCP Foundations.
   - **Containers/Kubernetes:** CIS Docker/Kubernetes Benchmark.
2. Document the chosen benchmark version in the System Security Plan (SSP) for each system.
3. Where CIS benchmarks are not available, substitute vendor hardening guides or DISA STIGs.

> **💡 Why you might choose differently:** CIS Level 2 is significantly more restrictive and may break functionality. For internal-only, low-sensitivity systems, Level 1 is usually sufficient. Choose Level 2 for internet-facing, Restricted-data, or high-compliance systems.

#### 1.2 Vendor Default Remediation

1. Before connecting any new system to the network, change ALL vendor defaults:
   - Default usernames (e.g., `admin`, `root`, `sa`, `Administrator`) — rename or disable.
   - Default passwords — generate unique random passwords using the password manager (`____`).
   - Default SNMP community strings (`public`, `private`) — disable or change.
   - Default TLS certificates — replace with organization-issued certificates.
   - Default management ports — move to non-standard where security-by-obscurity adds a layer.
2. Run an automated scan using `____` (e.g., Nessus, Qualys, OpenSCAP) against the system post-deployment to identify remaining defaults.

#### 1.3 Unnecessary Services and Protocols

1. Disable all services, ports, and protocols not required for the system's defined business purpose. At minimum, disable:
   - Telnet, FTP, TFTP (use SSH, SFTP instead).
   - SMBv1 (use SMBv3).
   - LLMNR, NetBIOS, WPAD (Windows).
   - rsh, rlogin, rexec (Linux/Unix).
   - SNMPv1/v2 (use SNMPv3 with auth+encryption, or disable).
   - HTTP for management interfaces (use HTTPS).
2. Validate with a port scan post-hardening; only approved ports should respond.

#### 1.4 Local Password and Account Controls

1. Enforce local account password policy via configuration management:
   - Minimum length: `____` characters.
   - Maximum age: `____` days.
   - Password history: remember last `____` passwords.
2. All local accounts must have passwords generated by and stored in the approved password management system (`____`). No human-known local passwords.
3. Rotate local account passwords automatically every `____` days via the password management system.
4. Disable guest accounts on all systems.
5. Rename or disable the built-in Administrator/root account where the system supports it; use named privileged accounts instead.

#### 1.5 Logging and Monitoring

1. Enable audit logging per the Logging and Monitoring Policy requirements.
2. At minimum, log: successful and failed authentication events, privileged account usage, security policy changes, system startup/shutdown, and file access to sensitive directories.
3. Configure log forwarding to the centralized SIEM (`____`) from all systems.

#### 1.6 Multi-Factor Authentication

1. Enable MFA for all interactive logins where the platform supports it.
2. For systems that don't natively support MFA, deploy a network-level MFA gateway or use SSH certificate-based authentication as a compensating control.
3. Document systems where MFA is not technically feasible in the risk register with compensating controls.

### Alternative Approaches

> **💡 Why you might choose differently:** Different security postures, compliance regimes, and operational scales call for different hardening philosophies.

- **Configuration-as-Code Automation:** Use tools like Ansible, Chef, Puppet, or cloud-native services (Azure Policy, AWS Config) to automatically apply and continuously enforce hardening baselines. Drift is automatically detected and remediated. Ideal for environments with >50 servers.
- **Golden Image Approach:** Build hardened VM/container images (AMIs, Docker images, VM templates) from which all systems are deployed. Images are rebuilt monthly with updated patches and hardening. No manual hardening post-deployment. Works well in immutable infrastructure / CI/CD environments.
- **CIS-CAT / SCAP Validation:** Use automated assessment tools (CIS-CAT Pro, OpenSCAP, Nessus compliance scans) to validate hardening post-deployment and on a recurring schedule. Generate compliance reports for auditor review. Score below `____`% triggers auto-remediation.

### Common Pitfalls

> **⚠️ Watch out:** "Set and forget" hardening. Systems drift over time as admins make tactical changes. Without continuous compliance monitoring, a hardened system today may be wide open in 6 months. Implement drift detection.

> **⚠️ Watch out:** Blindly applying CIS benchmarks without testing. Some settings break applications (e.g., service account restrictions, PowerShell execution policy, SELinux enforcing mode). Test in staging first; document and approve any deviations.

> **⚠️ Watch out:** Forgetting about cloud resources. S3 buckets with public access, security groups with 0.0.0.0/0 ingress, IAM roles with wildcard permissions — these are the #1 cloud breach vector. Use cloud security posture management (CSPM) tools.

## Procedure 2: Patching and Vulnerability Remediation

### Standard Approach

#### 2.1 Patch Identification and Evaluation

1. Subscribe to vendor security advisories for all in-scope software and hardware. Designate `____` as the vendor notification contact.
2. Monitor vulnerability intelligence sources: NVD (NIST National Vulnerability Database), CISA Known Exploited Vulnerabilities (KEV) catalog, vendor-specific advisories, and threat intelligence feeds.
3. On publication of a new advisory, evaluate within `____` business days:
   - CVSS score and severity (Critical ≥ 9.0, High ≥ 7.0, Medium ≥ 4.0).
   - Exploitability — is there a known public exploit? Is it actively exploited in the wild?
   - Exposure — is the vulnerable component present and reachable in our environment?
   - Mission impact — does the patch require a reboot or service interruption?
4. Assign remediation SLA based on severity:
   - **Critical (CVSS 9.0+) / Actively Exploited:** `____` days (e.g., 7-14).
   - **High (CVSS 7.0-8.9):** `____` days (e.g., 30).
   - **Medium (CVSS 4.0-6.9):** `____` days (e.g., 90).
   - **Low (CVSS <4.0):** Next scheduled maintenance window.

#### 2.2 Workstation and Server Patching

1. **Test Group:** Deploy patches to a test group of `____`% of workstations and non-production servers first. Monitor for `____` days for application compatibility, performance, and stability issues.
2. **Staged Rollout:** After successful test, deploy in stages:
   - Stage 1: `____`% of production workstations.
   - Stage 2: `____`% after `____` days with no issues.
   - Stage 3: Full deployment.
3. **Deployment Timing:** Schedule patch installation during defined maintenance windows. For end-user workstations, force install after a grace period of `____` days with user notifications.
4. **Compliance Monitoring:** Track patch compliance via the endpoint management platform. Systems out of compliance beyond SLA trigger an automated ticket and escalation to the system owner's manager at `____` days past due.

#### 2.3 Infrastructure Patching

1. Evaluate infrastructure patches (firewalls, switches, routers, hypervisors, storage arrays) as they become available.
2. Test in a lab or staging environment that mirrors production when a lab is available. For critical infrastructure where a lab isn't feasible, schedule during a low-traffic window with a defined rollback plan.
3. For redundant systems (clustered firewalls, HA pairs), patch one device at a time. Ensure failover works before patching the secondary.
4. Document all infrastructure patches in the change management system with: device, firmware version before/after, date, implementer, and verification results.

### Alternative Approaches

> **💡 Why you might choose differently:** Patch frequency and automation levels vary by organizational maturity, risk appetite, and operational constraints.

- **Automated Patching (Auto-Approval):** For low-risk, well-tested platforms (e.g., Chrome, Windows Defender definitions, browser extensions), configure fully automated patching without human approval. Reduces the "patch gap" to near zero for ubiquitous software.
- **Risk-Based Patching:** Instead of patching everything, focus resources on vulnerabilities that are exploitable in YOUR environment. Use threat intelligence and attack surface analysis to prioritize. Accept risk on low-exposure vulnerabilities to focus on high-impact ones.
- **Immutable Infrastructure:** Deploy new, fully patched instances and decommission old ones rather than patching in-place. Requires containerization/cloud-native architecture. Eliminates "configuration drift during patching" issues.

### Common Pitfalls

> **⚠️ Watch out:** Patch testing that takes longer than the SLA window. If testing takes 30 days and the SLA is 14 days, you're always out of compliance. Align testing capacity with SLA requirements or adopt canary deployment to accelerate testing.

> **⚠️ Watch out:** Forgetting about third-party applications. OS patching is easy; patching Adobe Reader, Java, Python libraries, Node.js packages, and Chrome extensions across an enterprise is harder. Use a software inventory tool that tracks third-party app versions and CVEs.

> **⚠️ Watch out:** "It's behind the firewall" mentality. Internal systems are not safe — lateral movement after initial compromise is standard attacker behavior. Apply the same SLA rigor to internal systems.

## Procedure 3: Physical Media Transfer

### Standard Approach

1. **Pre-Transfer Evaluation:** Before initiating a physical media transfer, validate that electronic transfer is not feasible. If electronic transfer IS feasible, prefer it — it's faster, more auditable, and eliminates physical chain-of-custody risks.
2. **Media Preparation:**
   - Use only organization-approved, FDE-capable external drives (approved models: `____`).
   - Encrypt all data on the media using the approved encryption standard (`____`, e.g., AES-256).
   - Verify encryption is applied before copying data: mount the encrypted volume and confirm contents are accessible only with the key.
3. **Courier Selection:**
   - Maintain an approved courier list managed by `____` (Facilities / Security). Vendors must pass a security assessment per the Vendor Management Policy.
   - Courier personnel must present photo ID at both pickup and delivery. The ID is verified against a pre-shared name.
4. **Packaging:**
   - Use tamper-evident packaging (sealed pouches, security tape).
   - Package must provide adequate physical protection — padded envelope for drives, rigid box for tape media.
   - Label with destination only; do NOT label with data classification or contents visible externally.
5. **Chain of Custody Logging:**
   - Log every transfer with the following fields:
     - Transfer ID (unique, auto-generated).
     - Description of contents (do not include actual data — e.g., "Q4 Financial Backup").
     - Data classification level.
     - Encryption method applied.
     - Date/time media handed to courier.
     - Courier company and courier personnel name (verified from ID).
     - Expected delivery date/time.
     - Recipient name and date/time of confirmed receipt.
   - Both parties sign (digitally or physically) the chain of custody at handoff and receipt.
6. **Receipt Verification:**
   - Recipient verifies tamper-evident seals are intact before accepting.
   - Recipient confirms contents are readable and complete.
   - Receipt confirmation is logged within `____` hours of delivery.

### Alternative Approaches

> **💡 Why you might choose differently:** The type of media, transfer frequency, and recipient location influence the best approach.

- **Encrypted Cloud Transfer:** Use an encrypted cloud storage intermediary (e.g., a time-limited, encrypted S3 bucket with presigned URLs) instead of physical courier. This is faster for long distances but requires both parties to have adequate bandwidth and security controls.
- **In-Person Courier by Employee:** For extremely sensitive transfers, an authorized employee carries the media personally. This eliminates third-party risk but introduces travel costs and availability constraints. Only suitable for infrequent, critical transfers.
- **Secure Data Room / Clean Room:** For recurring large-volume transfers, establish a dedicated secure transfer environment (physical or virtual) where both parties access the media under monitored conditions. Eliminates courier entirely.

### Common Pitfalls

> **⚠️ Watch out:** Assuming "the courier is FedEx, it's fine." Standard courier services do not guarantee chain of custody, and packages are lost routinely. Use specialized secure logistics providers for Restricted data.

> **⚠️ Watch out:** Encryption keys traveling with the media. If the password is on a sticky note in the same envelope, the encryption is worthless. Transmit keys through a separate channel (phone call, separate email, password manager sharing).

> **⚠️ Watch out:** Incomplete logs. "We shipped it Tuesday" is not defensible in an audit. Every transfer must have a complete, timestamped log entry with all fields.

## Procedure 4: Media Disposal and Sanitization

### Standard Approach

#### 4.1 Disposal Identification and Classification

1. The asset inventory system (`____`) must flag assets approaching end-of-life (EOL) or scheduled for replacement.
2. The Information Resource Owner confirms the asset is ready for disposal and identifies any data subject to retention holds (legal, regulatory, business).
3. Classify the disposal type:
   - **Reuse within organization:** Data must be sanitized; asset remains in inventory with new assignment.
   - **Return to vendor/lessor:** Data must be sanitized per vendor/lease agreement terms.
   - **Donation or sale:** Data must be sanitized; title transfer documented.
   - **Destruction:** Physical destruction of media; asset decommissioned from inventory.
   - **Damaged media:** Risk-assess for repair vs. destruction. If unreadable by normal means and contains Restricted data, destroy.

#### 4.2 Sanitization Methods

| Data Classification | Reuse / Return | Disposal / Destruction |
|--------------------|----------------|------------------------|
| **Restricted** | Cryptographic erasure + verified overwrite (NIST SP 800-88 Purge) | Physical destruction (shred, incinerate, degauss + shred) |
| **Confidential** | Cryptographic erasure or single-pass overwrite (NIST Clear) | Physical destruction or verified overwrite |
| **Internal** | Factory reset / standard wipe | Standard disposal (no special sanitization required) |
| **Public** | No sanitization required | Standard disposal |

**Sanitization Execution:**
1. For electronic sanitization, use approved tools (`____`, e.g., Blancco, DBAN for HDDs; platform-specific for SSDs — note that traditional overwrite is unreliable on SSDs).
2. Verify sanitization: for Purge-level, run a verification read to confirm all sectors contain the overwrite pattern. Document the verification result.
3. For physical destruction, use an approved destruction vendor (`____`) or in-house shredder rated for the media type (HDD shredder: `____` particle size). Witness destruction when possible.
4. Full-disk encryption (FDE) is a compensating control: if all media was encrypted from Day 1 with strong key management, cryptographic erasure (destroying the encryption key) is sufficient for Purge. This drastically simplifies disposal.

#### 4.3 Disposal Logging

For every disposal, log:
- Asset ID (from inventory).
- Media type and serial number.
- Data classification of stored data.
- Sanitization method applied.
- Verification result (pass/fail).
- Date of sanitization.
- Personnel performing sanitization (name + witness name if required).
- Final disposition (reused, returned, donated, destroyed).
- Destruction vendor name and certificate of destruction (if applicable).

Retain disposal logs for `____` years.

#### 4.4 Annual Sanitization Verification

1. Annually, select a random sample of `____` sanitized media (before destruction) and attempt forensic recovery using `____` (forensic tools).
2. If ANY data is recoverable from Purge-level sanitization, investigate root cause, re-sanitize all media processed with the same method since the last verification, and update procedures.
3. Document the annual test results for audit evidence.

### Alternative Approaches

> **💡 Why you might choose differently:** Volume, media types, and in-house capabilities dictate the best approach.

- **100% Encryption + Crypto Erase:** Encrypt ALL media from provisioning. Disposal becomes purely a key management exercise — delete the encryption key from the key management system, log the deletion, done. Eliminates physical sanitization entirely for most media. Requires rigorous key management infrastructure.
- **Third-Party Sanitization Service:** Outsource all disposal to a certified ITAD (IT Asset Disposition) vendor that provides on-site destruction and a certificate of destruction per media item. Ensure vendor meets NAID AAA or equivalent certification.
- **In-House Physical Destruction:** For high-security environments, install an in-house shredder. Media never leaves the facility until physically destroyed. Higher upfront cost, lower ongoing risk and chain-of-custody complexity.

### Common Pitfalls

> **⚠️ Watch out:** SSDs and overwrite. Traditional multi-pass overwrite tools designed for magnetic HDDs do NOT reliably sanitize SSDs due to wear-leveling and over-provisioning. Use ATA Secure Erase / NVMe Format commands, cryptographic erasure, or physical destruction for SSDs.

> **⚠️ Watch out:** "We'll sanitize it later" pile. Media accumulates in a storage closet with good intentions. The pile becomes a data breach waiting to happen. Implement a policy that media must be sanitized within `____` days of decommissioning.

> **⚠️ Watch out:** Cloud data disposal. Deleting an EBS volume in AWS doesn't guarantee the underlying physical media is sanitized. For high-security environments, require contractual commitments from cloud providers regarding media sanitization, or use customer-managed encryption keys that you can destroy.

> **⚠️ Watch out:** Copiers and printers with internal hard drives storing copies of every scanned/printed document. Include these in your asset inventory and sanitization procedures. Many organizations forget them entirely.
